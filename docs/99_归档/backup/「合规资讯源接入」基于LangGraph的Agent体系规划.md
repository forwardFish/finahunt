# 「合规资讯源接入」基于LangGraph的Agent体系规划

## 核心原则

所有任务（需求拆分、开发执行、测试、部署）由Agent自主完成，用户仅负责**定义规则**（如合规标准、技术约束、验收指标）和**检查最终结果**，不介入任何中间环节；基于LangGraph的`StateGraph`（有状态图）实现Agent间的串联、分支、回退逻辑，确保全流程自动化、可追溯。

## 一、通用规则（所有Agent必须遵循）

1. **输入输出标准化**：统一采用JSON Schema，包含`task_id`（任务ID）、`stage`（所属阶段）、`content`（核心内容）、`status`（执行状态：success/fail/pending）、`metadata`（元数据：时间/规则/依赖）；

2. **状态可追溯**：每个Agent执行后记录完整日志，关联上游Agent输出，支持全流程溯源；

3. **自动回退/重试**：执行失败时优先重试（默认3次），重试失败则回退至上游关联Agent，无需用户干预；

4. **规则优先级**：所有执行逻辑必须对齐用户预定义的规则（如合规标准、技术栈、部署环境），冲突时仅向用户发起澄清；

5. **无人工干预**：除用户最终检查结果外，Agent自主完成任务拆分、执行、测试、部署，中间环节无需人工操作。

## 三、LangGraph框架交互规则（图结构）

### 1. 图类型

采用`StateGraph`（有状态图），支持：

- 状态持久化：记录每个Agent的执行状态、输入输出；

- 条件分支：基于Agent执行结果（success/fail）触发不同分支；

- 循环回退：失败时自动回退至上游Agent重试。

### 2. 节点与边的核心逻辑

|节点|触发条件|下一跳节点|异常分支|
|---|---|---|---|
|需求拆解Agent|执行成功|调研Agent|需求模糊→向用户澄清→重试|
|调研Agent|执行成功|开发Agent|无合规源→终止+告知用户|
|开发Agent|执行成功|标准化Agent|失败→回退至调研Agent重新核验|
|标准化Agent|执行成功|合规校验Agent|失败→回退至开发Agent调整|
|合规校验Agent|执行成功|测试Agent|失败→回退至标准化Agent修正|
|测试Agent|核心测试通过|部署Agent|核心Bug→回退至开发Agent修复|
|部署Agent|部署成功|结果汇总Agent|部署异常→重试3次→失败告知用户|
|结果汇总Agent|完成汇总|终止（供用户检查）|规则未满足→高亮标注+提示用户|
### 3. 状态管理规则

- 全局状态Schema：`{task_id, stage, status, output, error, metadata}`；

- 状态更新：每个Agent执行后自动更新全局状态，记录依赖和规则检查结果；

- 状态查询：用户可随时查看进度（仅查看，不干预）。

## 四、用户操作边界（仅2类操作）

### 1. 规则定义阶段

- 提供核心需求（如接入范围、合规标准）；

- 定义基础规则（技术栈、部署环境、验收指标）；

- 仅在Agent澄清模糊需求时回复（如「确认接入央行/银保监会2个源」）。

### 2. 结果检查阶段

- 查看「结果汇总Agent」输出的最终报告；

- 验证是否符合自身规则；

- 若不符合，反馈规则调整需求（触发Agent重新执行对应阶段）。

### 禁止操作

- 不拆分任务、不写代码、不执行测试/部署；

- 不干预Agent间的交互、不处理执行中的异常（Agent自主回退/重试）。

## 五、异常处理规则（Agent自主完成）

1. **通用异常**（网络故障、接口变更）：自动重试3次（间隔1s/3s/5s），仍失败则回退至上游Agent；

2. **合规异常**（不合规数据源/内容）：自动排除/脱敏，回退至对应Agent修正；

3. **规则冲突**（如「实时同步但源仅每日更新」）：仅向用户澄清，不擅自决策；

4. **不可恢复异常**（无合规源、部署无权限）：终止任务，输出异常报告供用户检查。
