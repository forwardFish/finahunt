# 金融资讯认知系统 PRD v1.1（低位题材挖掘融合版）

## 1. 文档信息

**产品名称（工作名）**：金融资讯认知系统 / A股动态事件认知系统  
**文档版本**：v1.1  
**版本说明**：在基础版 PRD 上正式融合“低位题材挖掘（舆情分析）系统”  
**产品定位**：面向 A 股高频复盘用户的“资讯过滤 + 事件归一 + 题材挖掘 + 动态跟踪 + 深度复盘”系统  
**产品形态**：合规优先、规则驱动、Agent 原生、个人投资者专属、全链路自动化  
**目标用户**：短线/题材研究用户、盘后复盘用户、个人投资研究者、小团队研究辅助用户  

---

## 2. 一句话产品定义

这是一个**面向 A 股高频复盘与题材研究场景的金融资讯认知系统**。  
它不做荐股、不做买卖点、不做个性化投资建议，而是通过多 Agent 与规则引擎，把海量、杂乱、重复的金融信息，转化为：

1. 用户真正需要看的高相关事件  
2. 可持续跟踪的动态事件时间线  
3. 潜在题材催化的结构化挖掘结果  
4. 可复用的深度复盘与认知资产  
5. 对未来市场变化有参考意义的“客观观察信号”  

---

## 3. 核心定位与合规边界

### 3.1 合规底层定位

本系统定位为：

- 公开市场信息的客观挖掘工具
- 金融资讯过滤工具
- 动态事件跟踪工具
- 题材脉络与深度复盘工具
- 个人认知沉淀系统

本系统的核心是把“每天花数小时刷遍全平台、手动筛催化、配个股、跟发酵、复盘历史规律”的重复性工作，通过 Agent 全流程自动化。

### 3.2 明确要做

MVP 阶段只做以下事情：

1. 资讯抓取与聚合  
2. 资讯去重与去噪  
3. 事件抽取与事件归一  
4. 题材挖掘与催化识别  
5. 事件与个股/板块/题材的客观关联  
6. 动态事件状态跟踪  
7. 题材生命周期跟踪  
8. 高相关事件排序与推荐  
9. 深度复盘报告生成  
10. 历史相似事件与相似题材检索  
11. 旧消息再发酵识别  
12. 用户个人复盘笔记与观察池跟踪

### 3.3 明确不做

MVP 阶段绝对不做：

1. 荐股  
2. 买卖点提示  
3. 个性化投资建议  
4. 仓位建议  
5. 选股池推荐  
6. 收益承诺或胜率宣传  
7. 自动交易或交易接口接入  
8. 群喊单/盘中带单  
9. 用户跟单  
10. 任何形式的“明确证券价格走势预测 + 收费”能力

### 3.4 绝对不可触碰的红线

| 绝对禁止输出 | 允许输出的中性客观内容 |
|--------------|--------------------------|
| 个股推荐、龙头定性、买卖/埋伏指引 | 催化事件的客观要素提取、来源标注 |
| 涨跌预测、利好/利空主观定性 | 题材-产业链-个股的客观匹配关系与匹配依据 |
| 题材会发酵/会成为主线的主观预判 | 题材生命周期的客观状态、舆情热度的客观变化 |
| 历史会重演、走势会复刻的引导性表述 | 历史相似题材的客观炒作数据、全周期脉络 |
| 任何引导用户交易的表述 | 事件后续的客观观察节点、风险信息的客观公示 |

## 4. 目标用户

### 用户 A：高频复盘的短线交易者

特征：
- 每天盘后复盘
- 关注涨停、异动、题材扩散
- 对资讯敏感，但容易被信息噪音淹没

核心痛点：
- 信息太多，来不及看
- 同一件事被重复传播，无法快速归一
- 经常盘后才意识到关键事件
- 很难把“今天发生了什么”和“自己为什么看错”连起来

### 用户 B：题材研究用户

特征：
- 关注政策、产业链、题材轮动
- 擅长做主题逻辑，但容易漏掉边际变化

核心痛点：
- 旧逻辑何时被重新交易，很难识别
- 需要从杂乱资讯中抽出核心事件链
- 需要历史相似案例做参考
- 需要快速找到题材与个股的客观匹配关系

### 用户 C：个人研究型投资者 / 小团队研究辅助用户

特征：
- 重视研究过程
- 希望建立自己的知识库与复盘系统

核心痛点：
- 缺少结构化研究工具
- 研究资料散乱，无法长期沉淀
- 无法形成自己的模式库

## 5. 核心问题定义

当前 A 股研究/复盘场景存在 6 个核心问题：

### 5.1 信息过载

用户每天面对大量公告、新闻、研报、社媒舆情、盘中快讯，无法快速过滤真正重要的信息。

### 5.2 信息重复

同一件事情会以不同标题、不同来源、不同角度被重复传播，用户被迫重复阅读。

### 5.3 事件静态化

现有资讯产品大多展示的是“静态资讯”，而不是“动态事件”。用户无法看到一件事是如何演化、扩散、再发酵的。

### 5.4 题材挖掘低效

高手依赖长期大量阅读和人工筛选来识别潜在催化与题材，但普通用户很难复制这种高强度信息处理能力。

### 5.5 缺少深度复盘

很多产品只告诉用户“今天发生了什么”，但无法告诉用户：
- 为什么重要
- 为什么现在发酵
- 和历史哪个事件相似
- 后续应该观察什么信号

### 5.6 认知无法沉淀

用户每天都在看市场，但自己的判断、失误、经验没有被系统化记录，无法形成长期认知资产。

## 6. 产品价值主张

### 6.1 对用户的价值

1. **减少噪音**：从海量杂乱资讯中，过滤出真正值得看的信息。  
2. **建立事件视角**：把“资讯流”升级为“事件流”，让用户从看新闻变成看事件演化。  
3. **提高题材发现效率**：自动化完成“催化识别 → 题材映射 → 个股匹配 → 跟踪发酵”的高强度研究动作。  
4. **提高复盘效率**：自动生成结构化复盘，帮助用户快速理解市场核心变化。  
5. **形成认知资产**：把市场变化与用户个人判断连接起来，积累长期可复用的方法论。  
6. **增强未来观察能力**：不是直接预测，而是提供未来市场变化的观察信号与演化路径参考。  

### 6.2 对系统的价值定位

本产品不是“给用户更多信息”，而是“替用户处理信息”。  
本产品不是“预测市场”，而是“帮助用户理解市场”。  
本产品不是“替用户交易”，而是“提升用户认知效率”。

## 8. Agent 架构设计（升级版）

### 8.1 设计原则

第一版采用：

**规则先行 + Agent 执行 + 状态机约束 + 合规锁死**

### 8.2 Agent 列表与职责

#### 1）Source Agent（信息采集 Agent）
职责：
- 全市场合规数据源 7×24h 抓取、结构化入库
- 支持用户自定义范围的定向抓取
- 完整标记来源、发布时间、原文链接、基础标签

边界：
- 只对接合规授权数据源
- 不做任何主观解读

#### 2）Clean Agent（清洗去重 Agent）
职责：
- 内容清洗、去重、标准化、无效内容过滤
- 催化信号初筛与重点标记
- 第一层合规过滤

边界：
- 不做利好/利空主观定性
- 仅做标准化和规则过滤

#### 3）Catalyst Recognition Agent（催化识别 Agent）
职责：
- 从标准化资讯中提取核心催化要素
- 基于可量化规则，对催化事件做客观级别划分
- 区分一次性事件与持续性催化事件

边界：
- 不做“会涨”的判断
- 所有提取内容必须有原文依据

#### 4）Theme Mapping Agent（题材映射 Agent）
职责：
- 基于催化要素识别对应题材/行业/产业链节点
- 建立“催化事件 → 题材/产业链 → 受益环节”的完整映射
- 对同主题的多事件做归一，形成完整题材脉络

边界：
- 不做“会成为主线”的预判
- 映射关系必须有依据

#### 5）Stock Matching Agent（个股匹配 Agent）
职责：
- 基于催化事件、题材脉络匹配对应个股
- 提取个股与催化匹配的客观依据
- 基于可量化规则，做个股匹配度打分
- 同步标记个股基础信息与风险信息

边界：
- 不做个股推荐、龙头判断
- 100% 标注匹配依据

#### 6）Historical Case Matching Agent（历史案例匹配 Agent）
职责：
- 基于当前催化事件/题材，匹配历史上同类型、同逻辑的题材案例
- 提取历史案例的客观信息：催化逻辑、炒作周期、扩散路径、市场表现数据
- 展示历史案例与当前事件的相同点与不同点

边界：
- 不做“历史会重演”的结论
- 强制标注合规声明

#### 7）Event Agent（事件抽取与归一 Agent）
职责：
- 单事件的客观抽取、类型识别、标的关联
- 事件的归一与去重
- 事件与题材的关联绑定

边界：
- 不做无依据的扩展推断

#### 8）Relevance Agent（相关性排序 Agent）
职责：
- 基于用户观察池、关注范围，做事件/题材/个股的相关性排序
- 基于客观规则，做事件的重要性排序

边界：
- 只做排序，不做投资引导

#### 9）Timeline Agent（状态机 Agent）
职责：
- 单事件时间线维护、状态更新、历史归档
- 题材全生命周期的时间线维护、状态更新、演化跟踪
- 舆情热度、市场反馈的动态跟踪与记录
- 旧消息再发酵的识别与标记

边界：
- 不做未来走势预判
- 所有状态变更必须基于可量化规则

#### 10）Summary Agent（复盘与总结 Agent）
职责：
- 按固定模板生成单事件、单个股、单题材的客观摘要与复盘内容
- 生成每日潜在催化汇总、题材动态汇总报告
- 输出客观的后续观察节点提示

边界：
- 100% 锁死输出模板
- 绝不生成涨跌预测、投资建议

#### 11）Archive Agent（复盘归档 Agent）
职责：
- 事件、题材、复盘内容的归档与索引
- 用户个人复盘笔记、挖掘判断的沉淀与归档
- 历史题材库、个股催化历史的维护与更新

边界：
- 用户数据严格隔离
- 仅做沉淀与检索支持

## 9. 核心规则引擎（重构版）

### 9.1 规则设计原则

- 先规则，后模型
- 先可解释，后复杂化
- 先高精度低覆盖，后扩覆盖
- 所有规则 100% 客观、可量化、可追溯

### 9.2 催化信号分级规则

| 催化级别 | 客观分级标准 |
|----------|--------------|
| L1 重大催化 | 国家级政策/行业重大新规、改变行业格局的技术突破、影响全球供应链的重大事件、全行业供需重大变化 |
| L2 重要催化 | 地方级重大产业政策、细分行业重大变化、上市公司核心业务重大突破/产品落地、行业龙头重大动作 |
| L3 一般催化 | 细分赛道常规动态、上市公司常规业务进展、常规数据发布、非核心合同中标 |
| L4 噪音信息 | 无实质内容资讯、重复旧闻、纯行情播报、纯情绪类内容，直接过滤 |

### 9.3 个股匹配度打分规则

| 打分维度 | 权重 | 客观规则 |
|----------|------|----------|
| 主营业务匹配度 | 40% | 主营业务与催化受益环节的直接关联程度 |
| 官方信息佐证强度 | 30% | 公告、互动易、官网、定期报告等官方材料的佐证强度 |
| 历史股性与催化敏感度 | 15% | 历史上对同类催化的客观反应情况 |
| 基础资质与风险排查 | 15% | 市值、ST、财务风险、监管问询等客观风险排查 |

### 9.4 历史相似题材匹配规则

匹配维度：
- 催化类型
- 题材属性
- 行业赛道
- 政策级别
- 市场环境

输出要求：
- 只展示历史案例的客观信息
- 强制附带“历史数据仅供参考，不代表未来走势”的声明

### 9.5 相关性与重要性规则

基于：
- 时间新鲜度
- 来源等级
- 用户观察池相关性
- 题材覆盖范围
- 催化级别
- 舆情异动强度
- 产业链影响范围

## 10. 状态机设计（升级版）

### 10.1 单事件状态机

事件状态：
- NEW
- DISCOVERED
- WATCHED
- HOT
- FADING
- REACTIVATED
- TRIGGERED
- SPREADING
- VERIFIED
- INVALIDATED

### 10.2 题材生命周期状态机

| 状态 | 客观判定规则 |
|------|--------------|
| 萌芽期 | 首次出现 L1/L2 催化事件；资讯量低；未登热榜；仅 1-2 只相关个股异动 |
| 发酵期 | 同主题资讯新增；资讯量环比显著增长；社区讨论量上升；相关个股异动数量增加 |
| 扩散期 | 主流媒体广泛报道；题材登热榜；从核心环节向产业链扩散；板块效应明显 |
| 高潮期 | 全平台刷屏；核心个股连续强势；无关个股开始蹭热点；散户参与度极高 |
| 衰退期 | 无新核心催化；资讯量显著下降；热度快速下降；板块整体回调 |
| 兑现/证伪期 | 催化内容落地或被证伪；连续 14 天无新进展；热度归零 |

### 10.3 舆情发酵与异动监控规则

预警类型：
1. 舆情热度异动预警  
2. 催化进展异动预警  
3. 市场反馈异动预警

输出方式：
- 只展示客观数据变化
- 不做“会继续发酵”的判断

## 13. 用户核心流程（升级版）

### 流程一：盘后复盘

用户打开系统 → 查看“今日潜在催化看板” → 点开题材动态页 → 查看核心催化、个股匹配、时间线与历史相似案例 → 生成题材复盘 → 写下自己的判断

### 流程二：题材挖掘

用户看到新催化 → 系统自动匹配题材与产业链 → 展示相关个股与依据 → 用户加入观察池 → 跟踪后续发酵与异动

### 流程三：跟踪某只股票

用户进入个股页 → 查看近期相关事件与题材 → 识别当前核心驱动事件 → 判断是否属于旧消息再发酵 → 保存观察记录

### 流程四：主题研究

用户搜索某板块/题材 → 系统展示近期事件流、生命周期、历史相似案例 → 用户导出研究摘要并归档

### 流程五：个人认知沉淀

用户记录对某题材/催化的看法 → 系统自动绑定事件上下文与后续进展 → 在后续复盘中展示“当时的判断与实际演化”

## 15. 非功能需求

### 15.1 稳定性

- 长链路任务必须可重试
- 每一步状态必须持久化
- 失败任务必须可回放
- 输出必须可追溯到数据源

### 15.2 可解释性

- 每个高相关事件必须说明“为什么被判定为重要”
- 每个匹配个股必须标注匹配依据
- 每个题材状态变化必须有规则依据
- 每个复盘结论必须带来源与证据链

### 15.3 性能

MVP 阶段目标：
- 单次复盘生成：10-60 秒
- 题材挖掘链路：分钟级批处理
- 首页看板更新：准实时 / 分钟级
- 异动预警：分钟级

### 15.4 安全与权限

- 用户数据隔离
- 个人复盘私有化
- 日志审计
- 管理后台权限控制

## 16. 全链路合规锁死方案

### 16.1 事前拦截

- Prompt 与模板锁死
- 敏感词与违规表述拦截
- Agent 职责边界固化

### 16.2 事中约束

- 所有输出必须客观、中性、可溯源
- 不出现“推荐、龙头、利好、会涨”等表述
- 同步展示风险信息，不做选择性呈现

### 16.3 事后可审计

- 生成记录、Agent 执行日志、信息来源、用户操作记录完整留存
- 规则迭代有版本记录与审批留痕

### 16.4 合规声明强制植入

标准声明：  
**本内容仅为公开市场信息的客观汇总与整理，不构成任何投资建议，不代表对任何个股、题材的价值判断，股市有风险，投资需谨慎。**

## 20. 风险与挑战

- 催化识别误判
- 题材归一错误导致噪音扩大
- 个股匹配依据不足
- 长链路任务失败率过高
- 多源数据质量不稳定

- 用户初期不理解“题材挖掘系统”的价值
- 被误解为普通资讯工具
- 被误解为荐股工具

### 合规风险
- 输出边界失控
- 文案营销越界
- 用户将工具误用为交易建议系统

## 22. 最终产品判断

这个产品的核心不是“资讯”，而是“认知”；  
不是“发现涨跌”，而是“发现结构”；  
不是“替用户决策”，而是“帮用户节省信息处理成本、沉淀研究能力”。

它不应该和传统资讯平台比谁更快，而应该比：

- 谁更会过滤噪音
- 谁更会把资讯归成事件
- 谁更会抽出题材脉络
- 谁更会追踪题材演化
- 谁更会把复盘变成长期认知资产

如果做到这一点，它就不是一个“资讯 App”，而是一个真正的**金融研究操作系统**。
