# 设计切片与边界附录

本附录用于收拢原先零散的短文档，避免 `docs` 根目录堆积大量几百字的文件。后续如果某一节内容明显扩展，再从这里拆成独立文档。

## 1. 系统总纲

系统围绕“合规资讯源接入”构建，采用 Build、Runtime、Governance 三平面解耦架构，目标是让 Agent 原生执行、全链路可追溯、合规优先、关键节点可中断可恢复。

权威定义见 [Agent 体系设计文档 v1.0 正式落地版.md](../03_%E6%9E%B6%E6%9E%84/Agent%20%E4%BD%93%E7%B3%BB%E8%AE%BE%E8%AE%A1%E6%96%87%E6%A1%A3%20v1.0%20%E6%AD%A3%E5%BC%8F%E8%90%BD%E5%9C%B0%E7%89%88.md) 第 1、2 章。

## 2. Build Plane

职责是把需求落地为可在生产稳定运行的交付物。

节点：

- orchestrator
- requirement_parsing
- compliance_rules
- source_registry
- architecture
- contract
- development
- review
- test
- deploy_staging
- human_approval
- release
- build_summary

当前实现见 [build_graph.py](d:\lyh\agent\finahunt\graphs\build_graph.py) 和 [agents/build](d:\lyh\agent\finahunt\agents\build)。

## 3. Runtime Plane

职责是保障已交付能力的日常稳定运行，完成抓取、合规守门、标准化和审计。

节点：

- source_runtime
- compliance_guard
- normalize
- source_audit

当前实现见 [runtime_graph.py](d:\lyh\agent\finahunt\graphs\runtime_graph.py) 和 [agents/runtime](d:\lyh\agent\finahunt\agents\runtime)。

## 4. Governance Plane

职责是统一标准、执行策略、审计风险、量化评估，并为 Final Gate 提供决策依据。

节点：

- standards
- compliance_policy
- governance_audit
- evaluation

当前实现见 [governance_graph.py](d:\lyh\agent\finahunt\graphs\governance_graph.py) 和 [agents/governance](d:\lyh\agent\finahunt\agents\governance)。

## 5. Final Gate

职责是汇总三平面结果，输出唯一明确的最终决策。

输出要求：

- `final_gate_decision`
- `rule_check_matrix`
- `risk_notes`
- `action_recommendation`

当前实现见 [agent.py](d:\lyh\agent\finahunt\agents\final_gate\agent.py)。

## 6. Human Approval Checkpoint

职责是仅在生产发布前触发人工审批，支持审批、驳回、补充信息三类操作，并保留审计记录。

当前仓库默认采用自动通过策略以匹配当前执行要求，对应代码见 [checkpoint.py](d:\lyh\agent\finahunt\agents\approval\checkpoint.py)。

## 7. 人类角色边界

允许：

- 规则定义
- 关键审批
- 最终检查

禁止：

- 任务拆分
- 代码实现
- 测试执行
- 部署操作
- 日常排障
- 中间结果修改

## 8. Agent 职责边界

原则：

- 单一职责
- 结构化输入输出
- 可追溯
- 可阻断

校验点：

- 不允许跨职责产出结果
- 不允许越过合规与审批节点
- 不允许输出无版本规则结果

## 9. 三平面边界

Build Plane 负责 0 到 1 的建设与发布。

Runtime Plane 负责已交付功能的运行执行。

Governance Plane 负责标准、策略、审计、评估和放行支撑。
