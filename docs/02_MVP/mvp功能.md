   ## 一、MVP顶层核心定版（全周期不可动摇）

   ### 1. 一句话钉死MVP核心定位

   面向高频复盘的A股短线/题材研究用户，围绕用户观察池，完成**「资讯过滤→事件归一→动态跟踪→日终复盘」的最小闭环**，帮用户把2小时手动投研压缩到5分钟以内，先验证核心用户价值，再跑通商业闭环。

   ### 2. 全周期唯一北极星指标

   **用户每天在5分钟内完成「今日重点事件浏览+日终复盘」的完成比例**

   该指标直接对齐产品核心价值，同时约束过滤效率、排序合理性、页面体验、复盘可用性，是全阶段所有功能优化、规则迭代的核心锚点，比单纯的DAU、留存更贴合产品本质。

   ### 3. 全周期必须遵守的前置铁律

   1. **边界锁死铁律**：只保留「缺了就跑不通核心闭环、缺了就无法验证核心价值」的P0必做功能，所有非核心功能全部放入Beta迭代池，绝不进入MVP开发排期；
   2. **阶段递进铁律**：前一阶段验收不达标，绝不进入下一阶段，绝不提前开发后续阶段的功能，哪怕开发量极小；
   3. **合规优先铁律**：所有功能、所有输出内容，必须先过合规校验，合规是上线前提，不是后期补丁，绝不触碰证券监管红线；
   4. **极简至上铁律**：每个模块只做满足核心目标的最小功能集，不做冗余设计、不做复杂配置，把用户使用门槛和开发成本降到最低；
   5. **可迭代原则**：核心规则、权重参数仅设置初始默认值，MVP内测期可基于人工复核、用户反馈快速迭代，不写死不可调整的固定规则。

   ### 4. MVP功能边界与禁区（彻底解决冲突，明确分层）

   #### 【MVP仅做的最小差异化能力】

   仅做**旧消息重提的轻量规则化标签**：通过规则判断事件是否为历史事件重提，仅做状态标记和排序权重参考，不做复杂的再发酵原因分析、深度解读。

   #### 【永久禁区（MVP绝对不做，碰都不碰）】

   以下功能无论任何情况，都不进入MVP开发计划，全部留到MVP上线验证PMF后，再评估是否纳入Beta迭代：

   1. 非核心体验功能：用户笔记/收藏/标记看过、全局搜索、站内/站外通知推送、事件状态增强、用户可调规则、完整的旧消息再发酵识别、再发酵原因分析、历史相似案例匹配；
   2. 合规红线禁区：荐股、涨跌预测、买卖点提示、投资建议、收益承诺、行情/交易对接、任何引导投资决策的内容；
   3. 非核心产品功能：社区/UGC/互动功能、APP/小程序端开发（MVP仅做Web端）、用户自定义Agent/模板、复杂权限体系、独立个股详情页、独立板块/题材详情页、独立历史复盘归档页；
   4. 商业运营功能：复杂会员分层、分销/邀请/裂变体系、积分/成长体系、运营活动相关功能。

   ## 四、MVP分阶段落地执行全案

   严格按照「先验证引擎逻辑→再做产品化内测→最后跑商业闭环」的节奏拆分，适配单人开发的现实周期，每个阶段只聚焦解决一个核心问题，边界完全锁死，前一阶段不达标绝不进入下一阶段。

   ### 阶段一：核心引擎验证期（内部自用版）

   #### 核心定位

   完全剥离所有用户体系、前端界面、商业功能，只死磕最核心的投研自动化引擎，先让开发者自己每天能用，替代手动刷资讯、做复盘的工作，把核心逻辑、规则、准确率、合规性跑对跑稳，打牢整个项目的地基。

   #### 执行周期

   4-6周（适配单人开发节奏，预留规则优化、bug修复的缓冲时间，不赶激进进度）

   #### 核心目标

   1. 100%跑通「资讯接入→清洗去重→事件抽取→事件归一→排序→时间线管理→合规复盘生成」的完整核心链路；
   2. 输出标准化的事件对象和可直接使用的复盘内容，完全满足自用的每日复盘需求，无需再手动刷多个资讯平台；
   3. 把核心规则、合规约束、准确率打磨到可用标准，完成核心指标的验收达标，避免后期引擎大规模返工；
   4. 完成内部人工校正工具的开发，建立人工复核→规则优化的迭代闭环。

   #### 本阶段必做核心动作

   对应上文中「阶段一落地」的所有功能，核心聚焦5件事：

   1. 完成4类合规数据源的对接与测试，保证数据稳定、合规、结构化；
   2. 完成清洗去重、事件抽取、事件归一、排序、状态管理全链路Agent的开发与联调；
   3. 完成全链路合规拦截机制、内容分级生成能力的开发，守住合规底线；
   4. 完成底层日志、容错机制、内部人工校正工具的开发；
   5. 持续通过自用复盘、人工复核，优化核心规则、提升准确率，直到所有核心技术指标达到验收标准。

   #### 本阶段绝对不做的内容

   1. 不做任何用户体系、账号权限相关功能；
   2. 不做Web前端、可视化界面、用户交互功能，只做本地可运行的引擎，能输出标准化事件和复盘结果即可；
   3. 不做任何商业、付费、会员相关功能；
   4. 不做用户自定义配置、可调规则等非核心功能；
   5. 不做多用户适配、并发优化相关的开发。

   #### 硬性验收标准（不达标绝不进入下一阶段）

   1. 核心链路100%跑通，从触发抓取到生成完整复盘报告，全程自动化，无需人工干预；
   2. 所有核心技术指标，全部达到「核心指标验收协议」中的通过标准，连续2周稳定达标；
   3. 自用验证通过：每天用这套引擎生成的复盘，就能完成当日投研复盘，核心事件无遗漏、无错判，无需再额外刷3个以上的资讯平台；
   4. 合规与可信度达标：所有输出内容100%标注来源，无违规表述、无AI幻觉、无主观引导性判断，违规内容拦截率100%；
   5. 稳定性达标：引擎连续7天稳定运行，无崩溃、无核心链路失败，可每日自动生成复盘报告；
   6. 人工校正工具可用：可通过后台工具完成事件校正，校正结果可用于优化引擎规则。

   ### 阶段二：产品化内测期（种子用户可用版）

   #### 核心定位

   在阶段一已经跑稳的核心引擎基础上，做最小化的产品化封装，只做3个核心前端页面+极简用户体系，做成封闭内测版，让10-20个精准的目标用户能用上，先把「用户愿意用、留得住」这件事验证透，完全不碰商业付费。

   #### 执行周期

   4-6周（其中前2周完成产品化开发，后2-4周用于内测迭代、优化引擎、打磨体验）

   #### 核心目标

   1. 把自用的单用户引擎，封装成多用户可用的产品，跑通「用户注册→配置观察池→查看事件→查看复盘」的完整用户使用闭环；
   2. 让种子用户低成本上手，感受到核心价值，形成稳定的使用习惯，验证北极星指标和用户留存；
   3. 基于用户反馈、人工复核，持续优化引擎准确率、排序合理性、内容匹配度，完善产品体验。

   #### 本阶段必做核心动作

   对应上文中「阶段二落地」的所有功能，核心聚焦4件事：

   1. 完成极简用户账号体系、观察池/题材配置模块的开发；
   2. 完成3个核心前端页面的开发，实现用户侧的核心价值展示与交互；
   3. 完成核心引擎的多用户适配，优化定时任务体系，保证多用户场景下的稳定运行；
   4. 招募精准种子用户开启封闭内测，持续收集反馈，快速迭代优化引擎和产品体验。

   #### 本阶段绝对不做的内容

   1. 不做任何付费、会员、支付相关的商业功能；
   2. 不做笔记、收藏、搜索、通知推送等非核心体验功能；
   3. 不做用户自定义规则、可调参数功能；
   4. 不做APP、小程序端，只做Web端；
   5. 不做社区、UGC、分享裂变等运营类功能；
   6. 不做任何超出3个核心页面的额外前端开发。

   #### 硬性验收标准（不达标绝不进入下一阶段）

   1. 用户使用闭环100%跑通：从注册→配置观察池→查看事件→查看复盘报告，全流程顺畅无卡点，用户完成全流程平均耗时≤3分钟；
   2. 核心体验达标：用户能在3分钟内看完当日最相关的重要事件，能在系统内完成一次完整的盘后复盘，北极星指标达标率≥80%；
   3. 用户价值验证达标：10-20个精准种子用户，日均打开率≥80%，7日留存率≥60%，用户核心好评点集中在「省时间、不遗漏关键事件」；
   4. 系统稳定性达标：多用户场景下，单用户任务执行成功率≥99%，页面加载流畅，无核心功能bug；
   5. 合规达标：全平台内容100%符合合规要求，无违规内容、无引导性表述，所有内容可溯源，合规声明完整植入；
   6. 引擎优化达标：基于内测用户场景，核心技术指标持续稳定达标，无明显的错配、漏判问题。

   ### 阶段三：商业闭环上线期（正式可付费版）

   #### 核心定位

   在阶段二已经验证了用户价值、留存的基础上，只叠加最极简的商业闭环能力，完成可对外发布的正式MVP版本，验证用户付费意愿，跑通从「用户注册→使用→付费」的完整商业闭环，完成MVP的最终目标。

   #### 执行周期

   2-3周

   #### 核心目标

   1. 完成极简商业付费体系搭建，实现用户付费开通会员的完整流程，明确免费/付费权益边界；
   2. 上线正式可对外的MVP版本，完成从冷启动到付费转化的验证；
   3. 锁定MVP最终形态，不新增任何非商业必需的功能，只做bug修复和体验优化。

   #### 本阶段必做核心动作

   对应上文中「阶段三落地」的所有功能，核心聚焦3件事：

   1. 完成免费/付费权益体系、会员订阅模块、支付渠道对接、订单管理功能的开发与测试；
   2. 完成正式版上线前的全量合规审计、全链路压力测试、bug修复；
   3. 上线正式版本，开放对外注册，持续监控核心数据，验证付费转化。

   #### 本阶段绝对不做的内容

   1. 不新增任何核心功能，只做商业闭环必需的开发，不碰引擎核心逻辑、产品核心功能；
   2. 不做复杂的会员体系、多档位套餐、积分成长体系；
   3. 不做分销、邀请、裂变等运营类功能；
   4. 不做之前定的Beta阶段功能，全部留到MVP上线后再迭代。

   #### 硬性验收标准（MVP最终完成标志）

   1. 商业闭环100%跑通：用户从了解权益→发起支付→开通会员→解锁权益，全流程顺畅，支付成功率≥99%，权益实时生效，无卡点、无bug；
   2. 正式版可稳定对外开放注册使用，系统无核心故障，运行稳定，核心任务执行成功率≥99%；
   3. 商业价值验证达标：上线1个月内，种子用户付费转化率≥10%，付费用户月留存率≥60%，验证用户愿意为核心价值付费；
   4. 合规与安全达标：全流程符合监管要求，付费相关文案无夸大、无诱导，用户数据、支付信息安全合规，无合规风险。

   ## 五、最终执行铁律（全程必须严格遵守）

   1. **阶段边界绝对不突破**：每个阶段只做该阶段必做的功能，绝不提前开发后续阶段的内容，哪怕开发量很小，也坚决不碰，从根源避免需求蔓延、主线跑偏。
   2. **验收不通过绝不推进**：前一阶段的硬性验收标准没全部达标，绝不启动下一阶段的开发，避免在错误的基础上越走越远。
   3. **核心引擎优先原则**：所有资源优先投入核心引擎的打磨，只有引擎的准确率、稳定性、合规性达标，后续的产品化、商业化才有意义。
   4. **合规一票否决制**：任何功能、任何内容、任何文案，只要触碰合规红线，直接否决，无论对用户体验、商业转化有多大的增益，都绝不放行。
   5. **极简主义原则**：任何不在P0必做清单里的功能，一律先记录到迭代池，MVP阶段绝不讨论、不排期、不开发，先把最小闭环做透做极致。
   6. **快速迭代原则**：核心规则、排序权重、抽取逻辑，基于人工复核、用户反馈快速小步迭代，不追求一次完美，先保证可用，再持续优化。
   7. **人工兜底原则**：MVP阶段接受引擎的不完美，用后台人工校正工具做兜底，同时用人工标注的结果快速优化模型，形成正向迭代闭环。
