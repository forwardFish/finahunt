### 3. 「MVP的最小功能模块」：补充数据源合规底线，规避版权风险

### 4. 「MVP的Agent系统」：补充每个Agent的合规锁死规则，从底层规避风险

你定的6个Agent极简、完整，完全覆盖了核心闭环，没有任何冗余。重点补充每个Agent的**职责边界固化+合规约束**，避免开发中出现功能越界、合规漏洞，同时明确技术落地的适配性：

| Agent名称       | 固化唯一职责                                                | 必须落地的合规/边界约束                                      |
| :-------------- | :---------------------------------------------------------- | :----------------------------------------------------------- |
| Source Agent    | 只做用户关注范围的合规数据源抓取、结构化入库                | 只抓取用户观察池/关注题材相关的内容，不做全市场爬取；只对接合规授权数据源，不抓取无版权内容 |
| Clean Agent     | 只做内容清洗、去重、标准化、无效内容过滤                    | 增加第一层合规过滤，直接过滤标题党、虚假信息、无明确来源的内容；不做任何主观解读，只做内容清洗 |
| Event Agent     | 只做客观事件抽取、事件类型识别、标的/板块关联、事件状态判定 | 绝对不做「利好/利空」的主观定性，只做客观事实抽取；所有关联必须有明确的原文依据，不做无依据的关联延伸 |
| Relevance Agent | 只基于「和用户关注范围的匹配度」做事件排序打分              | 打分规则完全固化，只看「是否匹配用户观察池、是否是用户关注题材、事件重要等级」，不做任何涨跌相关的权重赋值 |
| Timeline Agent  | 只做事件的时间线维护、状态更新、历史关联归档                | 只做客观的事件演化记录，不做对未来的走势预判；所有状态变更必须有明确的信息触发，可追溯 |
| Summary Agent   | 只按固定模板生成客观的事件说明、复盘内容                    | 100%锁死输出模板，只允许在框架内填充客观事实、来源标注；强制植入合规声明，绝对禁止大模型自由生成主观判断、引导性内容 |

### 5. 「MVP的事件状态机」：补充固化的状态判定规则，避免AI幻觉

你定的5个状态极简够用，完全满足MVP需求，补充**可落地、可量化、非主观的判定规则**，避免大模型自由判断导致的状态混乱，同时保证可追溯：

| 事件状态  | 固化判定规则（100%可量化，无主观判断）                       |

## 三、MVP阶段必须死守的2条生死线

1. **合规生死线**：所有功能、所有输出，必须先过合规关，哪怕牺牲一些所谓的“吸引力”，也绝对不碰监管红线。所有Agent的输出，必须先过合规校验，再给到用户，没有例外。
2. **边界生死线**：所有不在MVP功能清单里的需求，一律不做、不讨论、不排期，哪怕用户提了，也先记录到后续迭代池里，先把核心闭环做透、做极致，再谈其他。用户的核心付费理由，是你把「事件过滤、归一、复盘」这件事做到了极致，而不是你做了多少功能。
