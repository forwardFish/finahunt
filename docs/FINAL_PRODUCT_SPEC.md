## 1. 一页结论

Finahunt 是一个面向 A 股个人投资者的合规型金融资讯认知系统。

它不做荐股、不做买卖点、不做涨跌预测、不做个性化投资建议，而是用 Agent、规则引擎和可审计数据链路，把分散、重复、噪声高的公开市场信息转化为可追溯、可复盘、可沉淀的事件认知资产。

第一阶段只打透一个高频刚需闭环：

> 用户设置观察池 / 关注题材 -> 系统接入合规公开资讯 -> 清洗去重 -> 抽取事件 -> 识别潜在催化 -> 聚合题材与关联对象 -> 按用户相关性排序 -> 生成结构化复盘与研究卡 -> 归档为用户认知资产。

当前最高优先级不是“展示更多资讯”，而是做深、做准、做可追溯：

> 潜在题材催化的结构化挖掘结果。

如果只能优先做一件事，就做这件事。所有页面、排序、复盘、预警、工作台，都必须围绕这个核心结果服务。

## 2. 产品定位

### 2.1 顶层定位

Finahunt 是国内面向 A 股个人投资者的“合规驱动、Agent 原生、专属投研信息处理与复盘自动化工具”。

它的本质不是资讯聚合，不是行情软件，不是交易系统，也不是 AI 荐股工具。它的本质是一个面向个人投研场景的事件认知操作系统。

### 2.2 长期使命

用 AI Agent 重构个人投资者的投研信息处理流程，让每一位 A 股个人投资者都能拥有专属、合规、高效的自动化投研助手。

核心目标不是替用户决策，而是替用户完成高重复、低效率、易遗漏的信息筛选、事件归一、逻辑梳理、风险提示和复盘归档工作。

### 2.3 长期愿景

3-5 年目标是成为国内个人投资者 AI Agent 投研赛道的基础设施平台。

长期形态从个人专属投研系统，逐步演进为投研 Skill、Agent、Workflow 的开放生态，重构个人投资者获取、理解、沉淀和复用市场信息的方式。

## 3. 核心用户

### 3.1 种子用户

第一阶段只服务一类高价值种子用户：

> 有 1 年以上 A 股短线 / 题材交易或研究经验，日均手动刷资讯时间不低于 2 小时，每天做盘前 / 盘后复盘，对事件时效性、关联性、证据链要求高，愿意为“省时间、不遗漏关键事件、复盘更系统”的工具付费。

### 3.2 用户分层

高频复盘用户：

- 每天盘后复盘。
- 关注涨停、异动、题材扩散和核心事件。
- 痛点是资讯太多、重复太多、关键事件容易遗漏。

题材研究用户：

- 关注政策、产业链、题材轮动和潜在催化。
- 痛点是旧逻辑何时再发酵难以识别，题材与个股关系需要大量人工梳理。

个人研究型投资者 / 小团队：

- 重视研究过程、复盘方法和历史沉淀。
- 痛点是研究资料散乱，历史判断无法结构化复用。

### 3.3 第一阶段不服务的人

- 只想看实时行情和交易信号的人。
- 需要明确荐股、买卖点或仓位建议的人。
- 没有复盘习惯、只想被动接收热点推送的人。
- 机构定制化客户。
- 需要全市场无差别资讯流的人。

## 4. 核心问题

Finahunt 解决六类问题：

1. 信息过载：公告、快讯、政策、新闻、社媒、研报摘要太多，用户无法快速筛出重点。
2. 信息重复：同一事件被多源、多标题、多角度反复传播，增加无效阅读成本。
3. 事件静态化：传统资讯产品展示新闻流，不展示事件如何演化、扩散、再发酵。
4. 题材发现低效：潜在催化、产业链影响、相关个股关系依赖人工筛选。
5. 复盘不够深入：用户知道“发生了什么”，但难以快速理解“为什么重要、后续观察什么、与历史哪个逻辑相似”。
6. 认知无法沉淀：用户每天的判断、笔记、错判、复盘没有结构化记录，无法长期复用。

## 5. 不可逾越的合规边界

合规是产品底层能力，不是页面底部的免责声明。

### 5.1 产品允许输出

- 公开市场信息的客观汇总与整理。
- 资讯清洗、去重、归一、结构化。
- 事件、题材、板块、个股之间的客观关系。
- 事件时间线、题材生命周期、历史相似案例的客观呈现。
- 风险信息提示、来源引用、证据链展示。
- 用户复盘笔记、观察池、研究归档和个人认知沉淀。
- “后续观察节点”，但只能是客观信号，不得变成操作建议。

### 5.2 产品绝对不输出

- 荐股。
- 买卖点提示。
- 个性化投资建议。
- 仓位建议。
- 收益承诺、胜率宣传、盈亏暗示。
- 明确涨跌预测。
- 自动交易或交易接口接入。
- 带单、跟单、社群喊单。
- “龙头确认”“必涨”“主线确认”“低吸机会”“买入观察”等引导交易的表达。
- 无差别全市场资讯推送。
- 未授权、不可商用、有版权风险的数据搬运。

### 5.3 标准合规声明

所有用户可见的研究输出必须附带或可展开查看：

> 本内容仅为公开市场信息的客观汇总与整理，不构成任何投资建议，不代表对任何个股、题材或市场走势的价值判断。股市有风险，投资需谨慎。

### 5.4 系统可信度三原则

所有核心输出必须满足：

1. 有来源：每个事件、数据、事实必须标注可追溯来源。
2. 有证据：每个关键结论必须对应原文片段、公告、新闻、数据或规则依据。
3. 有时间线：每个事件必须能还原演化过程，便于追踪和复盘。

## 6. 第一阶段业务闭环

### 6.1 最小闭环

第一阶段的业务闭环是：

1. 用户设置观察池 / 关注题材 / 关注标的。
2. 系统按合规源注册表抓取公开资讯。
3. 系统完成清洗、去重、标准化和来源标记。
4. 系统从资讯流中抽取事件，并合并多源重复事件。
5. 系统识别候选催化句、催化类型、强度和时效。
6. 系统将事件聚合为潜在题材，并映射产业链、板块和相关个股。
7. 系统基于用户观察池和规则计算相关性、重要性和研究优先级。
8. 系统生成今日重点、题材发酵、低位研究、复盘报告。
9. 用户记录判断、收藏事件、归档复盘。
10. 系统将历史判断与后续演化绑定，形成个人认知资产。

### 6.2 成功状态

用户打开产品后，不需要自己刷多个资讯源，而是能在 3 分钟内知道：

- 今天有哪些值得研究的核心事件。
- 哪些事件正在形成潜在题材催化。
- 催化属于什么类型、强度如何、时效如何。
- 对应哪些题材、板块、产业链环节和相关公司。
- 这些关联的证据来自哪里。
- 后续需要观察哪些客观信号。
- 自己过去是否看过类似事件，历史复盘如何。

## 7. MVP 范围

### 7.1 MVP 一句话

Finahunt MVP 要验证一件事：

> 能否把合规公开金融资讯压缩成用户可消费的事件流、题材线索、低位研究卡和日终复盘，让用户不再手工刷大量信息源。

### 7.2 P0 必做

P0 是第一阶段必须闭合的主线：

1. 合规资讯源注册表与接入规范。
2. 原始资讯、标准化资讯、事件对象、证据片段的统一数据契约。
3. 资讯清洗、去重、标准化、无效内容过滤。
4. 事件抽取、事件归一、事件类型识别。
5. 候选催化句抽取。
6. 催化类型识别。
7. 催化强度与时效性判断。
8. 潜在题材聚合。
9. 题材到板块、产业链、个股的客观映射。
10. 个股 / 公司匹配依据与风险提示。
11. 用户观察池相关性排序。
12. 今日重点事件、观察池事件、日终复盘输出。
13. 低位题材研究卡基础版。
14. 结构化结果仓库。
15. 运行 trace、gate registry、人工校正记录和 final gate。
16. 全链路合规治理与输出模板锁定。

### 7.3 P1 应做

P1 在 P0 稳定后推进：

1. 异动预警中心。
2. 历史相似案例基础版。
3. 研究归档与检索。
4. 个股详情页、题材详情页增强。
5. 轻量通知。
6. 旧消息再发酵识别。
7. 多视角研究工作台。

### 7.4 P2 暂缓

以下能力暂缓：

1. 复杂产业链知识图谱。
2. 用户自定义挖掘规则引擎。
3. 全量历史题材库深度建设。
4. 社区、分享、UGC、模板市场。
5. 复杂舆情情绪分析。
6. 多端 App / 小程序铺开。
7. 商业化系统的完整订阅、支付和权益闭环。

### 7.5 永久不做

以下能力不进入任何阶段：

1. 荐股和买卖建议。
2. 自动交易。
3. 仓位建议。
4. 收益承诺。
5. 非授权资讯爬取或版权风险内容商用。
6. 机构定制化外包式系统。

## 8. 数据源与数据治理

### 8.1 数据源原则

只接入合规、公开、可追溯、可审计的数据源。每个源必须在 Source Registry 中登记。

Source Registry 至少包含：

- source_id
- source_name
- source_type
- authority_level
- access_method
- license_or_usage_basis
- rate_limit
- robots_or_terms_constraint
- enabled_status
- last_reviewed_at
- risk_level

### 8.2 P0 数据源类型

- 财经快讯 / 电报类公开资讯。
- 沪深交易所公告。
- 上市公司公告与定期报告。
- 官方互动平台公开问答。
- 政策与监管部门公开文件。
- 主流权威财经媒体。
- 行业协会公告。
- 地方政府产业政策。

### 8.3 P1 / P2 数据源类型

P1 可补充：

- 产业链数据平台。
- 招投标公开信息。
- 专利公告。
- 更多垂直行业媒体。

P2 才考虑：

- 海外行业媒体。
- 跨境政策信息。
- 供应链公开信息。

### 8.4 禁止数据源

- 未授权付费研报全文。
- 私域社群聊天记录。
- 明确禁止抓取或商用的网站内容。
- 非公开数据。
- 用户未授权的个人交易数据。

## 10. 规则引擎

### 10.1 催化分级

L1 重大催化：

- 国家级政策。
- 行业重大新规。
- 改变行业格局的技术突破。
- 影响全局供需关系的重大事件。
- 全球供应链重大变化。

L2 重要催化：

- 地方重大产业政策。
- 细分行业重大变化。
- 上市公司核心业务突破。
- 产品落地或订单落地。
- 行业龙头重要动作。

L3 一般催化：

- 细分赛道常规动态。
- 上市公司常规业务进展。
- 常规数据发布。
- 非核心合同或常规合作。

L4 噪声信息：

- 无实质内容资讯。
- 重复旧闻。
- 纯行情播报。
- 纯情绪内容。
- 无证据支撑的市场传言。

### 10.2 催化类型

至少支持：

- 政策催化。
- 产业催化。
- 技术催化。
- 公司经营催化。
- 供需催化。
- 价格催化。
- 事件驱动催化。
- 监管 / 风险催化。
- 舆论热度催化。

### 10.3 个股匹配规则

个股匹配只能来自客观依据：

- 主营业务与催化受益环节的直接关联。
- 公告、互动易、官网、定期报告等官方材料佐证。
- 历史上对同类催化的客观反应。
- 产业链环节和业务收入占比。
- 市值、ST、财务、监管问询等风险信息。

匹配输出必须展示：

- 为什么相关。
- 证据来自哪里。
- 关联强度是多少。
- 有哪些风险。
- 为什么这不是投资建议。

### 10.4 正宗度 / 纯度评分

候选公司需要区分“真实相关”和“概念蹭边”。

评分维度：

- 主营业务直接性。
- 官方证据强度。
- 收入或产品落地程度。
- 产业链位置。
- 与催化事件的时间相关性。
- 风险与不确定性。

禁止输出“正宗龙头”“最受益”“确定性最高”等交易诱导表达。

### 10.5 低位研究优先级

低位题材研究优先级不等于买入建议，只代表“研究价值排序”。

评分维度：

- 催化强度。
- 题材热度处于早期还是后期。
- 共识程度是否尚低。
- 相关公司证据是否充分。
- 风险是否可识别。
- 用户观察池相关性。

输出必须命名为“研究优先级”或“观察优先级”，不得命名为“买入优先级”。

### 10.6 用户相关性排序

排序依据：

- 用户观察池对象。
- 用户关注题材。
- 用户历史复盘记录。
- 事件新鲜度。
- 来源权威等级。
- 催化等级。
- 证据完整度。
- 风险等级。

## 11. 状态机

### 11.1 单事件状态机

事件状态：

- NEW：新进入系统。
- DISCOVERED：已识别为有效事件。
- WATCHED：与用户观察池相关。
- HOT：热度显著上升。
- SPREADING：多源扩散。
- TRIGGERED：触发题材 / 板块 / 个股关联。
- REACTIVATED：旧消息再发酵。
- FADING：热度衰退。
- VERIFIED：证据确认。
- INVALIDATED：证据证伪或信息无效。

### 11.2 题材生命周期

题材状态：

- 萌芽期：首次出现 L1 / L2 催化，资讯量低，相关对象少。
- 发酵期：同主题资讯新增，讨论量或关联对象上升。
- 扩散期：主流媒体广泛报道，产业链或板块扩散明显。
- 高潮期：全平台刷屏，相关对象扩散过度，非核心对象也开始被提及。
- 衰退期：无新核心催化，热度下降。
- 兑现 / 证伪期：催化落地、被证伪或连续无进展。

状态变化必须基于客观数据和规则，不得由模型自由判断。

## 12. Agent 体系

### 12.1 产品运行 Agent

MVP 运行链路采用：

`Source -> Clean -> Event -> Catalyst -> Theme -> Candidate -> Relevance -> Timeline -> Summary -> Archive -> Compliance Guard`

Source Agent：

- 采集合规公开数据源。
- 严格遵守 Source Registry。
- 输出 RawContent。

Clean Agent：

- 清洗、去重、标准化。
- 抽取实体、关键词和证据片段。
- 输出 NormalizedContent。

Event Agent：

- 从标准化资讯中抽取事件。
- 合并多源重复事件。
- 维护事件状态。

Catalyst Agent：

- 抽取候选催化句。
- 判断催化类型、等级、时效性。
- 输出 Catalyst。

Theme Cluster Agent：

- 聚合潜在题材。
- 识别低共识题材簇。
- 维护题材核心叙事。

Candidate Mapper Agent：

- 将题材映射到板块、产业链、公司。
- 输出客观匹配关系。

Purity Judge Agent：

- 判断候选公司正宗度。
- 降权蹭概念、证据不足、风险过高对象。

Fermentation Monitor Agent：

- 跟踪题材热度、扩散、再发酵。
- 更新题材生命周期。

Similar Case Agent：

- 匹配历史相似题材。
- 展示相似点和差异点。
- 强制声明历史不代表未来。

Relevance Ranking Agent：

- 基于用户观察池和规则计算相关性。
- 输出排序结果流。

Review Writer / Summary Agent：

- 生成每日复盘、研究卡、事件摘要。
- 模板锁定，只填充客观事实和证据。

Archive Agent：

- 归档事件、题材、复盘、用户笔记。
- 支持后续检索和认知沉淀。

Compliance Guard：

- 对所有输出进行违规表达拦截。
- 记录合规结果和审计日志。

### 12.2 工程治理三平面

归档中的 Agent 架构文档提供三平面工程治理模型，应作为后续自动执行与平台化建设的标准。

Build Plane：

- 负责从需求到交付的开发建设。
- 包含 Requirement Parsing、Compliance Rules、Source Registry、Architecture、Contract、Development、Review、Test、Deploy Staging、Release、Build Summary 等 Agent。

Runtime Plane：

- 负责已交付功能 7x24h 稳定运行。
- 包含 Source Runtime、Source Compliance Guard、Normalize、Runtime Scheduler、Runtime Observer、Source Audit 等 Agent。

Governance Plane：

- 负责标准、合规、审计、评估和 Final Gate。
- 包含 Standards、Compliance Policy、Audit、Evaluation、Final Gate 等 Agent。

Human Approval Checkpoint：

- 人类只负责规则定义、关键审批、最终结果检查。
- 生产发布必须有人类审批。
- 其他普通执行环节由 Agent 自动完成。

### 12.3 Agent 输出规范

所有 Agent 输出必须结构化，禁止只输出自然语言长文。

统一输出至少包含：

- agent_name
- agent_version
- run_id
- trace_id
- status
- input_refs
- output_refs
- rule_version
- content
- evidence
- compliance_result
- errors
- created_at

## 13. 技术架构原则

### 13.1 总体原则

- 规则先行，模型辅助。
- 结构化对象优先，文本只是展示层。
- 合规前置，输出后置。
- 每一步可追溯、可重试、可回滚。
- 所有长链路任务必须有 checkpoint。

### 13.2 推荐实现

功能型 Agent：

- 使用 LangGraph 组织 Source、Clean、Event、Relevance、Timeline、Summary 等节点。
- 每个节点只做一件事。
- 使用全局 State 传递结构化对象。
- 在 Event、Relevance、Summary 设置人工校正点。

长链路编排：

- 可使用持久化工作流能力承载定时任务、重试、暂停、恢复和审计。
- 非确定性外部调用必须放入可追踪 Activity / Tool 中。

存储：

- 原始层不可变。
- 索引与关联层可增量更新。
- 每日服务层面向页面和 API 输出。

### 13.3 数据维护架构

运行数据分三层：

Immutable Raw Layer：

- 保存原始资讯、来源、抓取时间、原始哈希。
- 不直接修改，只追加。

Index and Association Layer：

- 保存事件、题材、公司、证据、相似案例、用户观察池的关联关系。
- 支持重算和版本化。

Daily Serving Layer：

- 面向页面和 API 的每日快照。
- 支持首页、题材页、研究页、工作台快速读取。

## 20. 风险与应对

### 20.1 合规风险

风险：输出被理解为荐股或交易建议。  
应对：模板锁定、敏感表达拦截、标准声明、人工抽检、审计日志。

风险：数据源版权或授权不清。  
应对：Source Registry 准入、授权状态记录、非白名单源阻断。

### 20.2 产品风险

风险：用户误解为普通资讯聚合。  
应对：所有页面围绕事件、题材、证据、复盘组织，不做无差别资讯流。

风险：范围膨胀。  
应对：以“观察池 / 关注题材 -> 事件过滤 -> 复盘归档”闭环作为优先级判断。

### 20.3 技术风险

风险：长链路任务失败率高。  
应对：checkpoint、重试、可恢复、错误分层、上游回退。

风险：事件归一和题材聚合误判。  
应对：证据片段、规则优先、人工校正点、历史回测。

风险：数据 contract 漂移。  
应对：Contract Agent、版本化、兼容性测试、Final Gate。

### 20.4 UI 风险

风险：页面变成信息堆叠。  
应对：四个页面职责分离，首页只做入口，工作台承接全量视图。

风险：中文乱码影响可信度。  
应对：文本编码检查、路由 HTML 检查、截图人工复核。

## 21. 当前研发优先级判断

任何新需求进入前，先问五个问题：

1. 是否服务“观察池 / 关注题材 -> 事件过滤 -> 复盘归档”最小闭环。
2. 是否强化“潜在题材催化结构化挖掘结果”。
3. 是否保持合规、来源、证据、时间线完整。
4. 是否减少用户手动筛选和复盘成本。
5. 是否避免提前扩张到交易、社区、开放生态和复杂预测。

不满足前四项、且提前触碰第五项的功能，默认不进入当前阶段。

## 22. 当前最终定义

Finahunt 的核心不是“资讯”，而是“认知”。

它不是发现涨跌，而是发现结构。

它不是替用户交易，而是替用户压缩信息处理成本、提高复盘质量、沉淀长期研究资产。

它不是和传统资讯平台比谁更快，而是比谁更能：

- 过滤噪声。
- 把资讯归成事件。
- 抽出题材脉络。
- 跟踪题材演化。
- 展示证据链。
- 记录用户判断。
- 把复盘变成长期认知资产。

做到这一点，Finahunt 才不是一个资讯 App，而是一个真正的金融研究操作系统。
